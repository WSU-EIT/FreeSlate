Email CSS

Rules for email-safe CSS

Email clients are a decade behind browsers. Follow these rules or your beautifully styled form confirmation will look like a ransom note in Outlook.

Inline styles only
No external stylesheets, no <style> blocks in the body (Gmail strips them). Every element gets its styles inline.
Font stack
Montserrat, 'Helvetica Neue', Helvetica, Arial, sans-serif — Montserrat renders on clients that load Google Fonts (most modern ones). Helvetica/Arial as bulletproof fallback.
Bulletproof buttons
Use <a> with display:inline-block; padding; background-color; border-radius — no <button>. For Outlook, wrap in a VML conditional comment.
Widths
Max 600px wrapper, percentage-based columns inside that (100% on mobile). Never use CSS max-width — use width on tables.
Images
Always include alt text. Many clients block images by default — the email must make sense text-only.
Dark mode
Add @media (prefers-color-scheme: dark) in a <style> tag in <head> only (head styles survive in most clients). Use color-scheme meta.
No Flexbox/Grid
Use tables for layout. Painful but real. The templates below use a single wrapper table.

In Slate Deliver: Paste the full HTML into your Deliver source template. Slate will merge fields like {{first_name}} at send time. Test with Litmus or Email on Acid before the first real send.
