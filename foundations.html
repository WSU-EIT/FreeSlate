00 · The plan

What this is, and how I'm approaching it

You've got the green light to set the rules. Right now every Slate form is hand-rolled HTML with no shared conventions. This kit replaces that with a single styling layer plus a library of copy-paste patterns. Here's the thinking I'm building on — tell me where to steer.

One source of truth. All styling lives in wsu-forms.css, driven by design tokens (CSS variables). Change the brand in one place and every form updates. It's scoped under .wsu-content — the class your build.xslt already puts on every form's <main> — so it drops in without fighting the existing WSU Web Design System.

Five principles

Native first
Real <label>, <fieldset>, <legend>, native inputs. No div-soup widgets — that's what keeps screen readers and AXE happy.
Accessible by default
Visible focus rings, 44px touch targets, AA/AAA contrast, error text tied to inputs with aria-describedby. A content editor following the pattern gets accessibility for free.
Copy-paste ready
Every pattern ships as a labeled live example plus the exact HTML to paste into Slate.
On-brand, not loud
Crimson #a60f2d + red #ca1237, Montserrat, the signature H1 bar. Restrained, per WSU's 70/20/10 color rule.
Forgiving
Sensible max-widths, spacing, and wrapping so forms look intentional even when someone just stacks fields without thinking about layout.
01 · Logistics

How this reaches your Slate instance

You mentioned you just edit the XSLT and "magic happens." Close — here's the precise path, so you know where the finished files land. I'll write a full deploy guide as the last step, but the short version:

The CSS
Upload wsu-forms.css to Database → Files → /shared/, then add one <link> to it in /shared/build.xslt's <head> — right after your existing stylesheets so it wins.
Cache-busting
Every time you edit it, bump the ?v=YYYYMMDD on that link or Slate serves the old cached copy for up to a day.
The patterns
Content editors paste the HTML snippets into a form's Edit HTML block (or a Custom Content field). Because the CSS is global, the markup just works.
Test first
A broken build.xslt takes down every external page. Do it in the test instance / /dev/ folder, then publish.

♿ Accessibility floor

This palette is AAA-corrected. Body text #262626 (13.5:1), help text #595959 (7:1), and crimson buttons #a60f2d with white text (7.4:1) all clear AAA. Error/success message text uses dedicated darker inks (#8a0019 / #155f30); the brighter shades are reserved for borders and icons.

The sleeper fix: input borders use --wsu-control-border #767676 (4.5:1), not the WDS default #b3b3b3 (2.3:1) which failed WCAG 1.4.11. Bright Red #ff0843 is display-only — never on text or controls.

⚠ Framework policy

Never load a CSS framework into a Slate page. Bootstrap and Tailwind ship global resets (Reboot/Preflight) that restyle every input, button, and table — they'll fight the WDS and this kit on every control. Font Awesome isn't present and shouldn't be added: its ::before icon mechanism collides with the kit's own (.wsuf-error, checkmarks, breadcrumbs). The standard: WDS + wsu-forms.css only; icons are inline SVG sized in em, colored with currentColor, aria-hidden unless meaningful.

Specificity budget: kit rules stay one class deep under .wsu-content. Only two !importants are sanctioned — the datepicker z-index and the reduced-motion kill — so per-form Edit Styles always remain the admin's escape hatch.

03 · Core controls (preview)

The field, dissected

This is the atom of the whole system. Label, optional help text above the control, control, and — when needed — an error message below. Help and error are tied to the input with aria-describedby so screen readers announce them.

Text field with help ✓ label + aria-describedby

We'll only use this to follow up about your application.

<div class="wsuf-field">
  <label for="email">Email address</label>
  <p class="wsuf-help" id="email-help">
    We'll only use this to follow up about your application.
  </p>
  <input type="email" id="email" name="email"
         autocomplete="email" aria-describedby="email-help">
</div>

Field states

Default, required, error, and disabled — all driven by a class on the wrapper plus the right ARIA. Click into the fields to see the focus ring.

The four states ✓ aria-invalid on error

Enter a 5-digit ZIP code.

<!-- Error state -->
<div class="wsuf-field wsuf-field--error">
  <label for="zip">ZIP code</label>
  <input type="text" id="zip" aria-invalid="true"
         aria-describedby="zip-err">
  <p class="wsuf-error" id="zip-err">Enter a 5-digit ZIP code.</p>
</div>

Choice & cards

Radio and checkbox groups wrapped in a <fieldset> with a <legend> — the accessible way to group choices. The card variant makes the whole tile clickable, great for "what kind of student are you?" decisions.

Buttons

Red by default, hovering to crimson with a soft lift. Secondary (outline) and tertiary (text) for lower-emphasis actions. Always real <button> elements with visible focus.

Using the system

All styling is scoped under .wsu-content — build.xslt puts that class on the form’s <main>, so kit classes work automatically.

Brand colors live in :root tokens (--wsu-crimson etc.). Retheme the whole system by editing the token block once.

Follow the 70/20/10 ratio: crimson/gray dominate, secondary 20%, accents (nature colors) ≤10% — never on buttons or text.
