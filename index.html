WSU EIT Forms — Showcase

Everything the kit can do on a Slate page, in one place: 108 paste-ready patterns, the official brand icon set, logo files, photography guidance, and the brand rules behind it all. Search, filter by tag, or jump by chapter. Every demo is live; expand Markup to paste and drop it into a Slate text block — no CSS, no scripts needed on the form.

New to this framework? The five conventions (2-minute read)

1 · Where things go in Slate. You'll touch four surfaces: the Instructions tool in the form builder (paste HTML blocks there), a field's help text (plain text, no markup), an event's Description tab, and a portal's static content blocks. Build questions with Slate's own field types — the kit themes raw builder output automatically.

2 · One class does the work. Components are a single wrapper class (wsu-eit-note, wsu-eit-cards, wsu-eit-table) around ordinary HTML. No nested class soup like the old system.

3 · The naming grammar. wsu-eit-badge = component · __kicker = piece of one · --done = variant, always paired with its base. Search any name in this page's box to see it used.

4 · Colors are tokens. Never type a hex value into content — the brand lives in build.css once. If something seems to need a new color, it doesn't; ask brand@lists.wsu.edu.

5 · Behaviors are opt-in. The few JS features activate only on data-eit-* attributes you add deliberately. No attribute, no behavior — and everything works without them.

Re-renders the examples as users with each condition see them — check that nothing relies on color alone (WCAG 1.4.1). Screen-reader views live on each card.

    ♿ Screen-reader-only context (.wsu-eit-sr)
