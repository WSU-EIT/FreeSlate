New to this framework? The five conventions (2-minute read)

1 · Where things go in Slate. You'll touch four surfaces: the Instructions tool in the form builder (paste HTML blocks there), a field's help text (plain text, no markup), an event's Description tab, and a portal's static content blocks. Build questions with Slate's own field types — the kit themes raw builder output automatically.

2 · One class does the work. Components are a single wrapper class (wsu-eit-note, wsu-eit-cards, wsu-eit-table) around ordinary HTML. No nested class soup like the old system.

3 · The naming grammar. wsu-eit-badge = component · __kicker = piece of one · --done = variant, always paired with its base. Search any name in this page's box to see it used.

4 · Colors are tokens. Never type a hex value into content — the brand lives in build.css once. If something seems to need a new color, it doesn't; ask brand@lists.wsu.edu.

5 · Behaviors are opt-in. The few JS features activate only on data-eit-* attributes you add deliberately. No attribute, no behavior — and everything works without them.

    Export — your override block

    How do I use this block?

    This is only the settings that differ from the shipped defaults — your changes, nothing else. To put them live:

    1. Keep one file — wsu-eit-overrides.css — and load it after build.css in build.xslt: 
      <link href="/shared/build.css" rel="stylesheet">
<link href="/shared/wsu-eit-overrides.css" rel="stylesheet">
    2. Paste this block into that file, replacing whatever was there, and bump its ?v= stamp. Done — every page adopts it.

    Why the second file wins: CSS is read top to bottom. When two rules set the same thing at the same strength — both target :root here — the one read last wins. build.css declares the brand defaults; your override file is read after it, so its values sit on top. You never delete a default, you just outvote it (that’s why each line shows its shipped: value in the comment). Nothing in build.css is touched.

    Don’t want a second file? Paste the same block straight into build.css under §TOKENS instead — it overwrites the defaults in place. The separate file is just easier to update and to undo.

    Import — match the sliders to a deployed block

    Paste an override block (or your whole wsu-eit-overrides.css) to set every control to what’s configured there.

    About this kit & WSU brand sources

    This kit restyles WSU Admissions’ externally-hosted Slate (Technolutions) pages — forms, events, portals — to match the live university brand. Every visual decision below traces to an official source: an exact quote with a link to the WSU page it appears on, or a direct quote from one of the uploaded reference docs. Requirements are mandatory brand/accessibility rules; recommendations are sanctioned guidance the kit deliberately follows.

    Brand guidelines — brand.wsu.edu (colors · typography · logos · imagery) and the Official WSU Colors & Typography Guide (PDF). Live styling reference — admission.wsu.edu and wsu.edu.     Color Required Crimson, gray & white are the foundation

    Crimson, gray, and white are the foundation of the WSU color palette.— brand.wsu.edu/colors

    In the kit: these are the only structural colors — crimson chrome, a gray text/scale ramp, white paper. Everything else is derived from them.

    Color Required The 70 / 20 / 10 ratio

    The palette should be 70% crimson and/or gray, 20% a secondary color, and 10% or less devoted to the accent color palette.— Official WSU Colors & Typography Guide (PDF), echoed at brand.wsu.edu/colors

    In the kit: a typical form reaches this automatically — white page, crimson chrome, and a sliver of accent on edges only. See Brand guidance for the ratio bar.

    Color Required Crimson is one exact value

    Crimson • PMS — 201 C • RGB — 166 15 45 • HEX — #A60F2D— Official WSU Colors & Typography Guide (PDF)

    In the kit: stored once as --wsu-eit-brand: #A60F2D in build.css §TOKENS; content never hand-types a hex. (Athletics’ #981E32 is a different palette — not used here.)

    Color Required Accents never act like a university color

    Do not use an accent color in a dominant way, implying that it is an official university color.— brand.wsu.edu/colors

    In the kit: accents (Autumn, Goldfinch, Vineyard, Pacific Sky, Midnight) appear only as ≤10% edges and surfaces — never as primary UI or body text.

    Color Required Don’t pair an accent beside crimson as an equal

    Do not pair an accent color with crimson in a way that implies it has equal or secondary status as a university color.— brand.wsu.edu/colors

    In the kit: accents are isolated to their own components; they’re never set as a second color against crimson chrome.

    Color Required No screened-back crimson, no crimson→gray gradient

    Do NOT screen back crimson or use it as a gradient with gray.— brand.wsu.edu/colors

    In the kit: the one sanctioned gradient is crimson→deep-crimson (wsu-eit-banner); crimson is always rendered at full strength.

    Color Recommended Use accents sparingly, with purpose

    Used strategically and sparingly, they can add visual interest, distinguish elements in a sequence, or differentiate parts of a whole.— brand.wsu.edu/colors

    In the kit: accents code stat bands and deadline highlights — sequence/“parts of a whole” uses, exactly as described.
