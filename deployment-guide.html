Files

What goes where in Slate

The kit is three CSS files. Upload them to Database → Files in the /shared/ folder. Keep the names exactly as listed — other admins and future-you will be confused if they don't match.

Always work in /dev/ first. The Branding Editor's Publish action copies /dev/ → /shared/. A broken build.xslt in /shared/ takes down every externally-hosted Slate page. The test instance only, until confirmed.

FileUpload toWhat it does
wsu-forms.css/shared/wsu-forms.cssThe form kit — all tokens, components, validation states.
wsu-chrome.css/shared/wsu-chrome.cssHeader, footer, banner, breadcrumbs, identity bar.
web-design-system-overrides.css/shared/web-design-system-overrides.cssExisting Slate overrides — don't replace, merge your changes in.

Edit build.xslt to link the CSS

Open /shared/build.xslt in the Branding Editor (or Files → edit). Find the <head> section and add the two new <link> tags after the existing web-design-system links so the kit wins the cascade:

<!-- existing lines — keep these -->
<link href="/shared/web-design-system/style.css?v=20220215" rel="stylesheet" />
<link href="/shared/web-design-system-overrides.css?v=20220215" rel="stylesheet" />

<!-- ADD THESE — WSU Form Kit -->
<link href="/shared/wsu-chrome.css?v=20260609" rel="stylesheet" />
<link href="/shared/wsu-forms.css?v=20260609" rel="stylesheet" />

Order matters. The kit must load after style.css and web-design-system-overrides.css or specificity fights will produce confusing results. Adding the kit last wins every cascade cleanly without needing !important.

Workflow

Dev → Publish safely

  1. Upload wsu-forms.css and wsu-chrome.css to /dev/, not /shared/.
  2. Edit /dev/build.xslt to reference the /dev/ paths temporarily.
  3. Preview a real form in the test instance at wst-2.test.technolutions.net. Run WAVE. Check on mobile.
  4. When satisfied: in Database → Branding Editor, click Publish. This copies everything in /dev/ to /shared/ atomically.
  5. Hard-refresh a live form (Cmd+Shift+R / Ctrl+Shift+R) to bypass browser cache and confirm the new styles appear.

Cache-busting with ?v=

Slate (and browsers) cache /shared/ CSS files aggressively — up to 24 hours in the browser, ~15 minutes on Slate's servers. Changing the file isn't enough: you must bump the ?v= datestamp in build.xslt or your changes will be invisible to most visitors.

<!-- Before edit: -->
<link href="/shared/wsu-forms.css?v=20260609" rel="stylesheet" />

<!-- After edit — bump to today's date: -->
<link href="/shared/wsu-forms.css?v=20260615" rel="stylesheet" />

Convention: YYYYMMDD. If you edit the file twice in one day, append a counter: ?v=20260615b.

Server cache
~15 minutes after publish — forms served from Slate's CDN layer
Browser cache
Up to 24 hours — the ?v= change busts it by making the URL unique
Hard refresh
Cmd+Shift+R (Mac) / Ctrl+Shift+R (Win) — bypasses browser cache without needing a ?v= bump

Per-form CSS overrides

For form-specific styles that shouldn't apply globally, use Form editor → Edit Scripts/Styles. Slate scopes this CSS inside #form_GUID_container automatically, so it only affects that one form.

/* Per-form override — e.g. wider fields for a specific form */
#form_abc123_container .wsuf-field {
  max-width: 100%;
}
#form_abc123_container h2 {
  color: var(--wsu-crimson);
}

Don't use !important in per-form CSS unless you're fighting Slate's framework CSS (not the kit). The kit is intentionally low-specificity so your per-form rules win cleanly.

Checks

Verify it worked

  • Open a hosted form in the test instance. Buttons should be WSU crimson, not the default Slate blue/gray.
  • Run WAVE on the form URL. Target: 0 errors.
  • In browser DevTools → Network → filter "CSS" — confirm wsu-forms.css loaded (status 200, not 304).
  • Check the ?v= in the loaded URL matches what you set in build.xslt.
  • Test on a real mobile device or Chrome DevTools at 375px width.
⚠ Gotchas

Pipeline landmines

⚠ One bad character = every page down

build.xslt is XSLT — an unescaped & or < pasted into it is a parse error site-wide (forms, app, portals all fail). Escape ampersands as &amp; in any URL. Edit in /dev/, validate, publish — never hand-edit /shared/build.xslt.

⚠ Absolute paths only

Assets must live in Slate Files and be referenced as https://…/shared/…. Relative URLs resolve differently between /register/, /portal/, and embeds; http assets get blocked as mixed content.

⚠ The Mobile Template key

Without the Mobile Template config key (Configuration Keys → Branding), mobile devices get Slate's default grey template that ignores your branding entirely — the kit is desktop-only in effect until it's set.

⚠ Typekit allowlist + GTM pile

If you ever switch to Proxima Nova, the Adobe Fonts kit only serves allowlisted domains — add wst-2.test… and futurecoug.wsu.edu or it silently falls back. build.xslt already injects 4 GTM containers + several pixels; don't add more per page (drags LCP, can shift layout).

Rollback

The Branding Editor keeps Version History for every file in /dev/. If a publish breaks something, go to Database → Branding Editor → Version History, select the last known-good version, and restore. The restore writes to /dev/; you still need to Publish to push it to /shared/.

Emergency: If build.xslt is broken and forms are down, revert immediately via Version History. Don't try to fix in /shared/ directly — use the Dev → Publish workflow even in an emergency.
